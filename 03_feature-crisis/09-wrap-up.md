## 回講:五張便利貼各對應什麼模式

不看前面的課,試著把下面五句話,各自用一句話講出「對應哪個模式、為什麼」:

1. 商品圖太糊要高清 → ?
2. 搜尋像壞掉 → ?
3. 客服想要 AI → ?
4. 老闆要報表 → ?
5. 到貨通知要即時 → ?

講完再對答案:

1. **商品圖**:大檔案不能塞資料庫或走 app 中轉 → **presigned URL** 直傳物件儲存,圖片交給 CDN。
2. **搜尋**:`LIKE '%關鍵字%'` 用不到一般索引 → **倒排索引**(**全文檢索**)反過來查「詞→文件」。
3. **AI 客服**:模型沒讀過自家文件會亂編 → **RAG**,先檢索再讓模型根據檢索結果回答。
4. **報表**:耗時工作同步等會逾時 → **長時任務模式**(先回應、背景做、再回報),再加**資料管線**把分析查詢挪出正式資料庫。
5. **到貨通知**:使用者要被主動告知 → **即時更新的決策**(polling/SSE/WebSocket 依方向性與頻率選)。

而貫穿這五個功能的地基,是 L1 教的:**先把 API 設計講清楚,新功能才有共同的門面**。

## 這個模組真正在練的直覺

不是要你背五個技術名詞,是練一件事:**拿到一個新需求,先問「這屬於哪一類老問題」,
再套對應的模式**。系統設計面試、真實工作裡的新功能,幾乎都能拆回這幾類老問題之一。

## 不看筆記的小考

**Q1.** 為什麼 `SELECT * FROM products WHERE description LIKE '%帆布包%'` 就算在
`description` 欄位加了 B-tree 索引,也不會變快?

**Q2.** Presigned URL 這個模式裡,app 伺服器到底做了什麼事、沒做什麼事?

**Q3.** 報表功能同時用到「長時任務」跟「資料管線」兩個模式,這兩個模式各自解決報表的哪一部分問題?

**Q4.** 什麼情境下你會選 WebSocket 而不是 SSE?請舉一個到貨通知以外的例子。

<details>
<summary>看答案前,先自己講一遍</summary>

**A1.** 萬用字元在開頭(`%帆布包%`)代表要找「描述中間任意位置」含這幾個字,
B-tree 索引的優勢建立在「排好序、從頭比對、每次砍半」,幫不上這種比對方式,
資料庫還是得整表掃描。

**A2.** app 伺服器只簽發一張「限時、限定檔名/動作」的通行證(presigned URL),
它**不搬運檔案本身**——實際的檔案傳輸是瀏覽器直接對物件儲存進行的。

**A3.** 「長時任務」解決的是「使用者不用為了一份要跑很久的報表卡住畫面」
(先回應、背景算、再通知);「資料管線」解決的是「報表查詢的資料要從哪裡來」
(不能對著正式資料庫硬跑,要有一份分析專用的資料副本)。兩者合起來才是完整的報表功能。

**A4.** 任何「使用者也要即時發東西給伺服器、且伺服器要即時轉給其他使用者」的場景都適合,
例如多人同時編輯同一份文件、或客服系統裡多個客服即時共同處理同一張工單。

</details>

## 深讀地圖

想把這個模組的每個主題再讀深一層,官方文件是最可靠的起點:

- Presigned URL:MinIO 官方文件「Presigned URLs」章節,或任一家雲廠商 S3 相容服務的
  「Uploading objects with presigned URLs」文件。

- RAG / 向量檢索:pgvector 官方 README(github.com/pgvector/pgvector)。
