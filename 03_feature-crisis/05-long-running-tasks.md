# L5|長時任務:報表不能讓使用者乾等 📖

🎯 這課結束時:你知道為什麼「同步等結果」在耗時工作上會出事,以及「先回應、背景做、再通知」這個模式怎麼運作。

📚 想深挖:先記住這個模式的形狀,實作細節會依你用的技術棧(語言、佇列系統)而不同,官方文件以你選用的工具為準。

## 合夥人的便利貼:我要報表

「幫我生一份上禮拜各分類銷售報表」聽起來像一句普通的 API 呼叫,工程師直覺寫法:

```
GET /reports/weekly-sales
```

前端呼叫,等回應,顯示報表。問題出在:這句查詢如果要掃過一整週的訂單、
按分類彙總、算金額,可能要跑十幾秒甚至幾分鐘 —— 而大多數瀏覽器與代理伺服器,
一個 HTTP 請求撐不到那麼久就會**逾時**(常見預設在 30 秒上下,依環境而異)。
使用者看到的不是報表,是一個轉圈圈之後跳出來的錯誤。

## 為什麼「同步等結果」在這裡行不通

一般 API(像 L1 的 `GET /products/123`)之所以能同步回應,是因為工作本身很快 ——
資料庫一次索引查詢通常是毫秒級。報表這類工作**耗時是分鐘級**,
硬要塞進「請求進來、算完、回應」這個同步模型,等於要求使用者的瀏覽器
跟中間所有網路設備都陪你等好幾分鐘,只要任何一環有逾時限制就會炸。

## 模式:先回應、背景做、再回報

正確的形狀是把「使用者要等的時間」跟「工作實際花的時間」拆開:

1. 使用者送出「幫我生報表」的請求
2. 伺服器**不做這件事**,只是把工作丟給背景 **worker**,立刻回應:
   「收到了,任務編號 `job-8823`,狀態:處理中」
3. 背景 worker 慢慢把報表算出來(可能花幾分鐘)
4. 使用者的介面用查詢或通知的方式知道「job-8823 完成了」,再去拿報表結果

前端從「發出請求 → 死等回應」,變成「發出請求 → 立刻拿到任務編號 → 之後再確認狀態」。
「之後再確認狀態」剛好就是上一課學的即時更新問題:可以用 polling
(每幾秒問一次 `GET /jobs/job-8823`),也可以用 SSE 推播完成通知,兩邊接得上。

## 跟模組二的佇列,同一個工具、不同的用途

模組二教過用佇列(**Queue**)削峰填谷:尖峰時把大量下單請求排隊,
避免瞬間壓垮資料庫,重點在**保護系統不被瞬間流量打垮**。

這裡用佇列的理由不一樣:報表任務**本來就很慢**,不管流量大不大都一樣慢,
重點在**把慢工作跟使用者的等待時間解耦**,讓使用者不用為了一件慢工作卡住整個畫面。
兩種用途都靠「請求先進佇列、worker 非同步處理」這個形狀,但要解決的問題不同 ——
一個是「量太大」,一個是「單件工作太慢」。

## 進度回報怎麼做

只回「處理中/完成」有時不夠,使用者想知道「還要多久」。常見做法是讓
worker 在處理過程中,定期把進度寫回一個大家都能查到的地方(例如任務狀態記錄裡的
`progress: 42%`),前端輪詢或訂閱時順便帶出這個數字。不需要精準到秒,
「大概跑到哪」就足夠安撫使用者不會覺得畫面卡死了。

## 失敗了怎麼辦(先點一下,模組四細談)

背景任務可能中途失敗:worker 當機、資料庫暫時連不上。
兩個關鍵詞先混個眼熟:**重試(retry)**——失敗後再試一次而不是直接放棄回報失敗;
**冪等(idempotent)**——同一個任務被重複執行兩次,結果也要一樣
(不能因為重試就把報表金額算兩次)。這兩件事怎麼做扎實,是模組四「可靠傳遞」那一課的主題,
這裡只需要知道:**長時任務的失敗處理,不能靠「應該不會失敗」的樂觀假設**。

## 收尾一問

如果有人建議「乾脆把 HTTP 逾時時間設成 10 分鐘,這樣就能同步等報表跑完」,
你會怎麼回應這個提議?它解決了問題,還是只是把問題往後延?

→ 下一課:報表算出來了,但老闆要的不只是一份週報 —— 資料要怎麼持續流向分析用的地方。資料管線。

## 📇 名詞卡

- **Worker(背景工作程序)** — 獨立於「接收使用者請求」的伺服器之外,專門執行耗時任務的程序。它從佇列或任務清單拿工作、慢慢處理,處理進度或結果寫回一個使用者端能查詢到的地方。
  - 想更深可以想想:概念上與作業系統的「背景程序」相通,實作依所選佇列系統(如 Celery、BullMQ 等)的官方文件而定。
- **佇列 (Queue)** — 先進先出的工作清單:請求先排進佇列,背景 worker 依序取出處理。可以用來削峰填谷(模組二),也可以用來把慢工作跟使用者等待時間解耦(本課)。
  - 想更深可以想想:模組二「佇列:把尖峰排隊掉」那一課有完整說明。
