## 為什麼「加功能」是系統設計問題

聽起來像五個獨立的小工程,但仔細看,每一張便利貼背後都藏著一個**經典的設計模式**:

- 「圖要更清楚」→ 檔案變大了,不能再塞進資料庫或讓自家伺服器中轉 —— 這是**大檔案儲存**的老問題。
- 「搜尋壞了」→ 使用者要的其實是資料庫索引解決不了的「模糊比對」—— 這是**搜尋引擎**的老問題。
- 「AI 客服」→ 模型不知道你家的退貨規則,亂編答案 —— 這是**先查資料再回答**的老問題。
- 「報表」→ 又大又慢的統計查詢,不能對著正式資料庫硬跑 —— 這是**分析管線**的老問題。
- 「即時通知」→ 使用者要主動被告知,而不是自己重整頁面 —— 這是**伺服器推播**的老問題。

這正是這一整個模組要練的直覺:**每個功能需求,先問「這屬於哪一類老問題」,
再套對應的模式去解**。選對模式,功能長出來就是健康的;
選錯模式硬幹(例如把報表查詢直接打在正式資料庫上,或把使用者照片存進資料庫欄位),
這個功能就是帶著**先天缺陷**出生的 —— 一開始能動,量一大就是下一次半夜三點的告警(模組四預告)。

還有一張隱藏的便利貼,其實是這五張的共同地基:**這些新功能,前端要怎麼跟後端講話?**
API 沒設計好,後面五個功能蓋起來都歪 —— 所以模組從這裡開始。

## 這個模組的成品

繼續用「好物市集」的 Docker lab,這次不是加武器打群架,是逐課長出新器官:

- **商品圖直傳**:MinIO(open source 的 **S3 相容**物件儲存)+ presigned URL,圖片交給模組一的 CDN
- **搜尋**:PostgreSQL 內建全文檢索,體驗「詞→文件」反過來查
- **即時通知**:一段會主動推播的連線
- **報表**:背景任務 + 一顆分析用的資料庫
- **AI 客服**:pgvector 存自家 FAQ,做一個最小可行的 **RAG** 客服

## 模組地圖

| 課 | 一句話 |

| L1 API 設計 | 功能的門面:前後端怎麼講好話 |

| L9 收尾 | 回講 + 小考 |

## 📇 名詞卡

- **S3 相容 (S3-compatible)** — 業界把 Amazon S3 的物件儲存 API 變成事實標準,很多開源/自架儲存系統(如 MinIO)照著同一套 API 實作,所以同一段程式碼幾乎不用改就能換底層。
  - 想更深可以想想:MinIO 官方文件首頁會直接講它與 S3 API 的相容範圍。
- **RAG (Retrieval-Augmented Generation)** — 讓語言模型回答前,先去「檢索」跟問題相關的自家資料,把找到的內容一起餵給模型,模型再根據這些真實資料生成答案 —— 而不是憑訓練記憶亂猜。
  - 想更深可以想想:本模組 L7 有完整動手示範。
