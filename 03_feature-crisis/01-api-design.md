# L1|API 設計:功能的門面 📖

🎯 這課結束時:你能看懂、也能設計一組「像樣」的 REST API,並知道什麼徵兆代表該換武器了。
🧩 需要先會:無,把 API 想成「前端跟後端講話用的一份合約」就夠了。
📚 想深挖:MDN「HTTP request methods」、「HTTP response status codes」;關鍵字:REST、idempotent、pagination。

## 為什麼從 API 開始,不是先寫功能

上一課的五張便利貼,每一張最終都要「前端」問「後端」要資料、或請後端做一件事。
這個問答的格式,就是 API(**Application Programming Interface**)。

API 沒設計好,代價不是難看,是**每加一個功能都要跟前端來回吵**:
欄位名稱不一致、同一件事有兩種呼叫方式、錯誤發生時前端不知道怎麼判斷。
先把「怎麼講話」的規矩定下來,後面五堂課蓋新功能才不會每次都吵架。

## REST 白話:把「東西」當名詞,把「動作」當動詞

**REST** 的核心想法很像圖書館的借書系統:每一本書是一個「資源」,
有一個固定的位置可以找到它(書架編號),你對它做的事只有借、還、續借幾種固定動作。

套進「好物市集」:

- **資源(名詞)**:商品是 `/products`,某個商品是 `/products/123`,
  某個商品的評論是 `/products/123/reviews`。
- **動作(HTTP 動詞)**:

| 動詞 | 意思 | 好物市集例子 |
|---|---|---|
| GET | 讀取,不改變任何東西 | GET /products/123(看商品詳情) |
| POST | 新增一筆 | POST /orders(建立新訂單) |
| PUT | 整筆覆蓋更新 | PUT /products/123(整包資料換掉) |
| PATCH | 局部更新 | PATCH /products/123(只改價格) |
| DELETE | 刪除 | DELETE /products/123(下架商品) |

**壞設計**長這樣:`POST /doAction` body 裡塞 `{"type": "create_order", ...}` ——
所有動作擠進同一個入口,前端要看 body 才知道發生什麼事,伺服器端也難分權限與流量。

**好設計**是資源與動詞分開:`POST /orders` 一看就懂「這是在建訂單」,
不用開 body 猜意圖,權限、流量限制、快取規則都能按資源設定。

## 狀態碼:講清楚「結果」

前端不該用「猜」的方式知道成不成功,常用的這幾個要記熟:

| 狀態碼 | 意思 | 什麼時候用 |
|---|---|---|
| 200 | 成功 | 一般 GET/PATCH 成功 |
| 201 | 建立成功 | POST 新增一筆之後 |
| 204 | 成功但沒內容 | DELETE 成功、不用回傳東西 |
| 400 | 你的請求本身有問題 | 缺欄位、格式錯 |
| 401 / 403 | 沒登入 / 沒權限 | 分清楚「你是誰」跟「你能做什麼」 |
| 404 | 資源不存在 | /products/999999 沒這個商品 |
| 409 | 衝突 | 例如模組二講過的併發下單搶到同一件庫存 |
| 500 | 伺服器自己出錯 | 不是使用者的錯,是我們的錯 |

原則:**狀態碼負責「這件事的結果分類」,body 裡的訊息負責「細節」**,
兩者都省一個,前端都要多寫一堆 if-else 去猜。

## 分頁:清單類 API 一定要有

`/products` 這種清單,商品一多,一次全撈回來會拖垮資料庫也拖垮前端。
最簡單的方式是 `?limit=20&offset=40`(要 20 筆,從第 40 筆之後開始) ——
好懂,但資料一直在變動時,offset 分頁可能漏資料或重複。
規模大一點會改用「**cursor 分頁**」:回傳一個指標(通常是上一筆的 id 或時間戳),
下一頁請求帶著這個指標往後翻,不受中途插入資料影響。
「好物市集」現階段,offset 分頁夠用;量體變大再換。

## 什麼徵兆表示 REST 不夠用

REST 不是萬靈丹,兩個常見訊號:

- **前端一頁要拼七、八個請求**(例如訂單詳情頁要商品、庫存、評論、賣家資料分別呼叫) ——
  這時候值得看一眼 **GraphQL**:讓前端一次描述「我要哪些欄位」,伺服器一次組好回傳。
- **內部服務跟服務之間高頻互相呼叫**(不是給瀏覽器用,是機器對機器) ——
  這時候值得看一眼 **gRPC**:效能更好、有明確的介面定義,但學習成本比 REST 高。

## 原則:對外先 REST,別過早換武器

「好物市集」現在的規模,前端頁面請求數不多、也沒有大量內部微服務互相呼叫。
GraphQL 跟 gRPC 都是解決「規模到了之後」的特定痛點,提早導入只是多一層學習成本
換不到對應的好處。**先把 REST 做扎實,痛點真的出現、量得出來,再考慮換。**

## 📇 名詞卡

- **API (Application Programming Interface)** — 前端(或任何呼叫方)跟後端講話用的一份約定:哪個網址代表哪個資源、要傳什麼欄位、會收到什麼回應。API 設計得好,新功能加起來就不用每次重新吵格式。
  - 想更深可以想想:MDN:Introduction to APIs。
- **REST (Representational State Transfer)** — 一種設計 API 的風格:把資料看成「資源」給固定網址,把「動作」對應到 HTTP 動詞(GET/POST/PUT/PATCH/DELETE),讓 API 的形狀可預測。
  - 想更深可以想想:MDN:HTTP request methods。
- **GraphQL** — 另一種 API 風格:前端在一次請求裡自己描述「我要哪些欄位」,伺服器一次組好回傳,適合避免一頁要拼好幾個請求的情況。
  - 想更深可以想想:graphql.org → Learn。
- **gRPC** — 適合機器對機器、高頻呼叫場景的通訊框架,用明確定義的介面(protobuf)取代 JSON,效能較好但學習與除錯成本較高,通常不直接暴露給瀏覽器用。
  - 想更深可以想想:grpc.io → Introduction。
- **Cursor 分頁** — 分頁時不用「第幾筆到第幾筆」的位移量,而是帶著「上一筆的位置指標」往後翻,資料中途新增或刪除也不會漏頁或重複,大型清單常用。
  - 想更深可以想想:任一雲端 API 文件的 pagination 章節通常會對比 offset 與 cursor 兩種做法。
