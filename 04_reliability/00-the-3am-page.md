## 一年後的好物市集

爆紅那晚已經是一年多前的事了。「好物市集」撐過了流量爆炸(索引、快取、讀取複本、CDN 都上了),撐過了週年慶秒殺(交易鎖、佇列、分片都補齊了),也長出了你能想到的功能:站內搜尋、商品圖直傳、即時通知、報表管線,甚至還有一個會查自家資料的 AI 客服。

阿哲很得意:「這次真的穩了吧?」

三個月後的一個凌晨,他的手機在床頭震動。

## 二十分鐘,倒了整站

以下時間都是事後從各種紀錄兜出來的 —— 事發當下**沒有人看得到這條時間軸**,這點很重要,先記著。

**凌晨 03:00** — 排程照例跑「熱門商品清單」的每日重算工作,這個批次工作要對主資料庫做幾條掃過大量訂單紀錄的統計查詢。平常這工作安安靜靜跑個幾分鐘就結束。

**03:02** — 這次不巧,批次工作跟一小波時差關係還醒著的海外訪客撞在一起,主資料庫同時要應付「大量統計查詢」加「平常的線上流量」,那幾條統計查詢從平常的幾百毫秒拖到要跑好幾秒。

**03:04** — 首頁的「猜你喜歡」區塊,背後是推薦服務,推薦服務跟那些變慢的查詢共用同一個資料庫連線池(**connection pool**)。查詢一慢,推薦服務手上的連線就一直卡在「等資料庫回話」,推薦服務的平均回應時間從 50 毫秒飆到 4 秒。

**03:06** — 首頁服務呼叫推薦服務時**沒有設定逾時(**timeout**)**,於是每一個「使用者打開首頁」的請求,都乖乖排隊等推薦服務回話。沒多久,首頁服務自己對外能用的連線也全部卡在這個「等待」上,一個都不剩。

**03:09** — 連線被佔滿之後,連跟推薦完全無關的「看商品詳情」「看購物車」這些原本很快的請求,也一起分不到連線、跟著逾時 —— 一個非核心功能,拖垮了整個首頁服務。

**03:12** — 使用者開始瘋狂重新整理網頁;系統內部服務之間互相呼叫失敗時,本來就設計了「自動重試」,這下疊加使用者手動重試,湧進的請求量不減反增。

**03:15** — 請求持續堆積,連結帳(付款)流程都開始出錯,有些訂單卡在「處理中」動彈不得。

**03:17** — 一條很粗糙的告警規則(「整站錯誤率超過 50%」)終於被觸發,手機震動,阿哲被吵醒。他打開電腦,螢幕上只有一片重複的「500 內部錯誤」,完全看不出兇手是哪一段 —— 只能一個服務一個服務手動重啟,四十分鐘後才恢復。

## 復盤:三個補不完的洞

隔天的復盤會(**事後復盤**)上,大家發現真正可怕的不是那句慢查詢,是慢查詢背後**三個一直沒被解決的問題**,任何一個補上,那晚都不會這麼慘:

1. **看不見** —— 事發時沒有任何一塊儀表板告訴你「現在慢在哪一段」,阿哲是靠猜、靠手動重啟一台一台試出來的。等他真正意識到出事,已經是整站癱瘓之後。
2. **擋不住** —— 推薦服務只是個「錦上添花」的功能,卻沒有任何保險絲把它跟「看商品」「結帳」這些核心功能隔開,一個服務慢,拖著全站一起沉。
3. **對不上帳** —— 事故當中,系統重啟、請求重試交錯發生,有些訂單的通知或扣庫存訊息重複處理了兩次,也有些憑空消失,事後對帳兜不起來。

這個模組,就是把這三個洞一個一個補起來 —— 順便也把「這些服務到底該怎麼部署、住在哪裡」這件事想清楚。

## 模組地圖

| 課 | 一句話 |
|---|---|
| L1 可觀測性 | 補上「看不見」:logs、metrics、traces 三件套 |
| L2 過載保護 | 補上「擋不住」:給系統裝三種保險絲 |
| L3 可靠傳遞 | 補上「對不上帳」:訊息會丟也會重複,怎麼辦 |
| L4 你的程式住在哪 | container、serverless、API gateway 一次講清 |
| L5 放手課 | 回到這次事故,親手給架構裝上保險絲 |
| L6 收尾 | 回講這次事故 + 全系列(500 人到百萬人)總複習 |

## 收尾一問

時間軸裡,「一個非核心功能拖垮整個首頁服務」發生在哪一步?用你自己的話說說看,為什麼「共用同一個連線池」是問題的關鍵?

→ 下一課:先把「看不見」這個洞補起來 —— 不裝儀表板,你就是在盲飛。

## 📇 名詞卡

- **Connection Pool(連線池)** — 服務不會每次都重新跟資料庫建立連線(很貴),而是預先開好一批連線放在池子裡重複借用。池子裡的連線一旦被慢查詢全部佔住,後面的請求就只能排隊,這正是本課故事第 03:06 步的兇手。
  - 想更深可以想想:關鍵字:database connection pooling(多數資料庫驅動與框架文件都有專章)。
- **Timeout(逾時)** — 「等你回話最多等 X 秒,超過就當你失敗」的設定。沒設逾時,呼叫方會被慢下游無限期拖住;這是本課故事第 03:04→03:06 步能被擋下的第一個開關,模組四第二課會細講。
  - 想更深可以想想:關鍵字:request timeout、client timeout。
- **Postmortem(事後復盤)** — 事故結束後,團隊坐下來釐清「發生了什麼、為什麼、怎麼避免重演」的會議與文件。好的復盤不抓戰犯(blameless),目標是改流程,不是罵人。
  - 想更深可以想想:公開關鍵字:blameless postmortem(Google SRE Book 有專章)。
