# L4|你的程式住在哪:container、serverless、API gateway 📖

🎯 這課結束時:你能講清楚 container 跟 serverless 的差別、怎麼選,也知道 API gateway 在架構裡扮演什麼角色、跟負載平衡器怎麼分工。
🧩 需要先會:模組一的負載平衡器;這整個系列的 lab 你其實已經在用 **Container** 了。
📚 想深挖:關鍵字 container orchestration、Kubernetes(官方文件 concepts 章節)、serverless computing、API gateway pattern(microservices.io 有公開說明)。

## 你其實已經用過 container 了

這個系列的每一個 lab,你都是打 `docker compose up` 把「好物市集」跑起來的。現在正式介紹它到底是什麼:**container(容器)**把程式、程式需要的函式庫、執行環境全部打包成一個「貨櫃」,不管在你的筆電還是雲端的伺服器,打開這個貨櫃跑出來的行為都一模一樣 —— 不再有「我電腦上明明可以跑」這種事。

服務一多,總得有人負責「幫我開幾個貨櫃、哪個貨櫃掛了自動重開、流量變大自動多開幾個」—— 這是**容器編排器**(如 Kubernetes)在做的事,這課先知道有這麼一層就好,不深入。

## Serverless:連機器都外包出去

Container 還是需要「一台機器來跑貨櫃」。**Serverless** 更進一步:你只上傳程式碼,平台在真的有請求進來時才幫你生一個執行環境去跑,跑完就回收 —— 你只為「真正執行的時間」付錢,沒有流量的時候幾乎零成本,也不用自己顧著伺服器死活。

代價是兩個:

- **冷啟動(cold start)**:好一陣子沒人呼叫,平台把執行環境回收了,下一個請求進來要先重新生出環境,會多等一下。
- **執行時間通常有上限**:平台不歡迎你佔用資源太久,不適合本來就該長時間跑的任務(呼應模組三的長時任務,那種活通常還是要靠常駐的背景工作程式)。

## 怎麼選

沒有標準答案,但有一個實用的判斷:

- **流量穩定、持續**(首頁、結帳這種核心服務)→ 用 container 常駐,反而比較划算,也不用忍受冷啟動。
- **流量零星、突發、有明顯離峰**(深夜批次工作、圖片轉檔、接第三方系統的 webhook)→ 用 serverless,沒事發生時不用付錢。

回頭看 L0 的事故:那個凌晨 03:00 跑的「熱門商品重算」批次工作,其實就是很適合 serverless 的候選 —— 但更關鍵的是不該讓它跟線上流量共用同一個資料庫連線池,這是下一課放手課要處理的架構問題。

## API Gateway:所有外部請求的統一大門

服務一多 —— 首頁、推薦、搜尋、結帳、AI 客服各自成一個服務 —— 不能讓每個服務各自面對外面的網路世界,不然身分驗證、限流的邏輯要重複寫十遍。****API Gateway**** 是所有外部請求進來的統一大門:先在這裡做身分驗證、限流(呼應上一課的 rate limit)、決定「這個請求該轉給哪個內部服務」,內部服務只要專心做自己的事。

跟模組一學過的負載平衡器怎麼分工?負載平衡器負責「同一種服務有好幾台,這個請求該轉給哪一台」;API gateway 負責「不同種服務,這個請求該進哪一道門」。很多現成產品其實把兩者做在一起,但概念上是兩層不同的決策。

## 📇 名詞卡

- **Container(容器)** — 把程式、需要的函式庫、執行環境打包成一個可攜的「貨櫃」,在任何裝了容器執行環境的機器上跑出來的行為都一樣。這系列的 lab 一直用 docker compose 起這種容器。
  - 想更深可以想想:關鍵字:OCI container、Docker 官方文件 Get Started。
- **API Gateway** — 所有外部請求進來的統一大門,負責身分驗證、限流、把請求轉發給正確的內部服務,讓各個內部服務不用各自重複實作這些邏輯。
  - 想更深可以想想:關鍵字:API gateway pattern、microservices.io。
