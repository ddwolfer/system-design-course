# L4|佇列:把尖峰排隊掉 🔨

🎯 這課結束時:你知道秒殺開賣那一瞬間,為什麼「先讓十萬個請求排隊」比「讓十萬個請求直接衝資料庫」更聰明,並且能親手做出一個最小的排隊系統。
🧩 需要先會:L2 的原子 UPDATE(佇列的消費者最終還是要靠它安全地扣庫存)。
📚 想深挖:任一訊息佇列(如 RabbitMQ 或 Kafka)的官方入門文件;關鍵字:message queue、producer/consumer、backpressure。

## 秒殺開賣那一秒,發生了什麼

L2、L3 解決了「兩個人同時搶同一件貨,結果要正確」——但秒殺當晚真正恐怖的
不是「正不正確」,是**數量**。晚上八點整,十萬個人在同一秒按下購買鍵,
十萬個請求同時衝向資料庫。就算每一個都用了 L2 的原子 UPDATE 保證不超賣,
資料庫本身能同時處理的連線數與寫入速度還是有限——十萬個請求同時擠進門,
資料庫要嘛被打爆,要嘛把大部分人晾在那裡等到逾時。

## 削峰填谷:別讓洪水直接衝進城

想像一座水庫:上游暴雨瞬間沖下大量的水,水庫不會讓洪水直接灌進下游城市,
而是先攔住,再用下游河道能承受的速度慢慢放水。佇列(queue)做的正是這件事:

- 十萬個下單請求先全部**排進佇列**(這一步很快,幾乎不會塞車)。
- 一批**消費者(consumer)**按照資料庫真正扛得住的速度,一個一個從佇列裡
  拿出請求來處理、真正去扣庫存。
- 尖峰的洪水被攤平成資料庫能消化的細水流——這就是 **削峰填谷**。

## 非同步的使用者體驗:先給收據,再給結果

排隊系統帶來一個必須交代清楚的改變:使用者按下購買鍵的當下,
訂單**還沒有**真的被處理完,只是被放進了隊伍。介面該做的是:

1. 使用者送出訂單,系統立刻回覆:「已收到,搶購中,請稍等結果」
   (絕對不能讓畫面卡在轉圈圈,那會讓人以為當機了)。
2. 消費者在背景依序處理,真正決定「搶到了」還是「賣完了」。
3. 結果出來後,通知使用者(頁面輪詢、或事後跳通知/寄信——
   即時通知的做法,模組三會細講)。

這叫 **非同步 (asynchronous)** 處理:「接受請求」和「完成請求」
被拆成兩個時間點,中間允許有延遲。跟 L2、L3 的即時(同步)處理不同,
使用者要有心理準備:「搶到」不再是按下去那一秒就知道,而是排隊後才揭曉。

## 動手:一個最小的排隊系統(Redis list 示範)

用 Redis 的 list 當一個最陽春的佇列(`LPUSH` 推進去、`BRPOP` 排隊取出來)。
生產環境會用 Kafka 或 RabbitMQ 這類專職的訊息佇列(有持久化、重試、
多消費者分組等機制),但排隊的核心概念完全一樣:

```python
import redis, uuid

r = redis.Redis()
QUEUE = "order-queue"

def submit_order(product_id):
    """使用者按下購買:立刻回一個收據,真正處理留給背景"""
    request_id = str(uuid.uuid4())
    r.hset(f"order-result:{request_id}", "status", "queued")
    r.lpush(QUEUE, f"{request_id}:{product_id}")
    return request_id   # 前端拿這個 id 去查詢後續結果

def worker():
    """消費者:按自己(資料庫)能承受的速度,一個一個處理"""
    while True:
        item = r.brpop(QUEUE, timeout=5)      # 沒有訂單就乖乖等,不是死迴圈硬撐
        if item is None:
            continue
        _, payload = item
        request_id, product_id = payload.decode().split(":")
        changed = deduct_stock(product_id)     # L2 學到的原子 UPDATE
        status = "success" if changed else "sold_out"
        r.hset(f"order-result:{request_id}", "status", status)
```

`submit_order` 幾乎瞬間回應(只是塞進一個 list),不管尖峰多恐怖都塞得進去;
`worker` 則按照你開幾個消費者、資料庫扛不扛得住,決定實際處理的節奏。

## 什麼時候不該用佇列

佇列犧牲的是「立即性」,換來的是「扛得住尖峰」。所以只適合**寫入類、
可以晚一點才有結果也沒關係**的動作(下單、寄信、產生報表)。
使用者要「馬上看到答案」的**讀取**動作——例如查詢商品還有沒有貨、
登入驗證——不該排隊,那些請求該走模組一教的武器(索引、快取),
讓資料庫直接、快速地回答,排隊反而製造出不必要的等待。

## 常見坑

| 症狀 | 原因 |
|---|---|
| 使用者以為下單失敗,因為畫面沒有馬上顯示「成功」 | 沒做好非同步 UX——提交當下一定要給明確回饋,不能讓畫面卡住空等最終結果 |
| 佇列越堆越高,永遠處理不完 | 消費者處理速度長期低於湧入速度,得加開更多消費者,或在源頭做流量管制(如排隊頁面) |
| 伺服器重啟後,排隊中的訂單憑空消失 | 用了不持久化的陽春佇列;正式環境要選有持久化與重送機制的訊息佇列 |
| 把「查詢庫存」也塞進佇列 | 讀取需要馬上見分曉,不該排隊;只有寫入且能接受延遲的動作才適合 |

## 收尾一問

有人擔心:「加了佇列,使用者是不是要等更久了?」
分別從「使用者體感的等待時間」和「資料庫承受的實際流量」兩個角度回答他。

→ 下一課:鎖與佇列都撐住了,但如果訂單量已經大到**一台資料庫寫入到極限**,
下一步只剩一個選項:把資料庫本身切開。

## 📇 名詞卡

- **削峰填谷 (Peak Shaving)** — 把瞬間湧入的尖峰流量,透過佇列先攔住、再用系統能承受的穩定速度慢慢消化,而不是讓尖峰直接衝擊資料庫。
  - 想更深可以想想:關鍵字:queueing theory、backpressure。
- **非同步處理 (Asynchronous Processing)** — 把「接受請求」和「完成請求」拆成兩個時間點,中間允許延遲——使用者先拿到一張「處理中」的收據,結果稍後才揭曉,而不是全程等在原地。
  - 想更深可以想想:對照組是「同步(synchronous)」處理,像 L2 的原子 UPDATE 那樣按下去馬上見分曉。
