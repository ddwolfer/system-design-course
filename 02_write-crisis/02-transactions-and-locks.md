## 先把 ACID 講白話

「交易(transaction)」四個字的保證,教科書叫 ACID,這課你只需要真的搞懂第一個字:

- **A,Atomicity(原子性)**:一個交易裡的所有動作,**要嘛全部發生,要嘛完全沒發生**,
  不會發生「做到一半」的中間狀態被別人看見。這是這一課的主角。
- **C,Consistency(一致性)**:交易做完,資料要符合你定的規則(例如庫存不能是負的)。
- **I,Isolation(隔離性)**:多個交易同時跑,彼此不該互相干擾——但「隔離到什麼程度」
  其實有好幾個等級,細節留給深挖層。
- **D,Durability(持久性)**:一旦交易說「完成了(commit)」,資料就真的寫進硬碟,
  就算下一秒斷電也不會消失。

## 為什麼「包進 transaction」還不夠

很多人以為只要把 L1 那段「讀庫存 → 判斷 → 扣庫存」包進一個
`BEGIN ... COMMIT` 就安全了。**不夠**——原因很直接:

```sql
BEGIN;
SELECT stock FROM products WHERE id = 42;   -- 讀
-- 中間你的程式在判斷、算邏輯……
UPDATE products SET stock = 9 WHERE id = 42; -- 寫(把算好的結果寫回去)
COMMIT;
```

這兩句 SQL 中間,一樣有 L1 那個「空隙」——只是空隙從 Python 變數搬進了資料庫。
另一個交易一樣可以在你的 `SELECT` 和 `UPDATE` 之間插進來,做一模一樣的事,
兩邊都以為自己讀到「還有貨」。**光是 transaction 這個框,不會自動幫你堵住空隙**,
要嘛讓空隙消失,要嘛在空隙上鎖。三種修法分別對應這兩個方向。

## 修法一:一句原子 UPDATE(讓空隙消失)

最簡單也常常最好的解法:把「檢查」和「動作」寫進**同一句** SQL,不要分成兩步:

```sql
UPDATE products SET stock = stock - 1
WHERE id = 42 AND stock > 0;
```

這句話的威力在於:「庫存夠不夠」的檢查,和「扣庫存」的動作,對資料庫來說是
**同一個不可分割的操作**——外面沒有任何空隙讓別的交易插隊。你的程式接著看
「這句話改了幾列(affected rows)」:改了 1 列,代表搶到了;改了 0 列,
代表庫存不夠,乾脆地告訴使用者賣完了。

## 修法二:悲觀鎖(SELECT ... FOR UPDATE)

如果邏輯比「扣一個數字」複雜(例如要先算折扣、查會員等級再決定扣多少),
沒辦法濃縮成一句 UPDATE,就用 **悲觀鎖**——**先把這一列鎖起來,
鎖著的時候誰都不能碰**:

```sql
BEGIN;
SELECT stock FROM products WHERE id = 42 FOR UPDATE;  -- 鎖住這一列
-- 程式在這裡安心地做任何複雜判斷,不怕別人插隊
UPDATE products SET stock = stock - 1 WHERE id = 42;
COMMIT;  -- 提交的瞬間才放鎖
```

叫它「悲觀」是因為它假設「一定會有人來搶」,所以先把門鎖上再說——
另一個交易對同一列下 `FOR UPDATE` 會**乖乖排隊等**,直到第一個交易 COMMIT。

## 修法三:樂觀鎖(版本欄位)

如果大部分時候根本不會撞在一起(衝突很少見),悲觀鎖的「先鎖再說」就是浪費——
**樂觀鎖**反過來:**先假設不會撞,事後才檢查有沒有撞**。做法是加一欄
`version`:

```sql
-- ① 先讀出目前的庫存與版本號
SELECT stock, version FROM products WHERE id = 42;
-- 假設讀到 stock=5, version=7

-- ② 更新時,把「讀到的版本號」也寫進 WHERE 條件
UPDATE products SET stock = 4, version = 8
WHERE id = 42 AND version = 7;
```

如果這段時間沒人動過這一列,`version` 還是 7,這句話改到 1 列,成功。
如果有人搶先一步改過(version 已經變成 8),這句話改到 **0 列**——
你的程式發現改了 0 列,就知道「剛剛被人插隊了」,重新讀一次最新資料再試一次。

## 悲觀 vs 樂觀,怎麼選

| 情境 | 建議 |
|---|---|
| 秒殺、搶票這種高衝突場景(一堆人搶同一件) | 悲觀鎖——反正一定會撞,不如先排好隊,省掉大量白做工的重試 |
| 一般網站的日常下單(衝突機率很低) | 樂觀鎖——大部分請求一次就成功,不必每次都付鎖的代價 |
| 邏輯就是「扣一個數字」這麼簡單 | 優先選修法一,原子 UPDATE 最簡單也最快 |

**判斷的關鍵只有一個:衝突頻率。** 衝不衝得到,決定該先鎖著等,還是先衝再說。

## 動手驗證

把 L1 的腳本改成連到真正的資料庫(用 `psycopg2` 或你熟悉的套件),讓 50 個
執行緒對 `products` 表的同一列做「讀庫存 → 判斷 → 扣庫存」兩句分開的 SQL——
你會看到跟 L1 一模一樣的超賣,只是這次發生在真正的資料庫裡。接著把扣庫存的
邏輯換成修法一的原子 UPDATE,重跑 50 個執行緒,確認:成交筆數等於庫存數,
最終庫存停在 0,一件不多也不少。

## 常見坑

| 症狀 | 原因 |
|---|---|
| 用了 SELECT FOR UPDATE,還是超賣 | 忘記把讀與寫包在同一個 transaction 裡——鎖只在 transaction 存活期間有效 |
| 樂觀鎖一直重試失敗 | 衝突其實很頻繁,選錯武器了,該換悲觀鎖 |
| 原子 UPDATE 改了 0 列卻沒處理 | 程式沒檢查 affected rows,誤以為都成功,使用者以為買到卻沒扣到貨 |
| 悲觀鎖鎖住整張表 | 忘記在 WHERE 裡指定 id,FOR UPDATE 鎖到不該鎖的列,拖慢無關的訂單 |

## 📇 名詞卡

- **悲觀鎖 (Pessimistic Lock)** — 先假設一定會有人來搶,動手前就把資料鎖起來,其他人得排隊等鎖放開。適合衝突頻繁的場景,代價是大家要等。
  - 想更深可以想想:PostgreSQL 文件:Explicit Locking 章節,SELECT ... FOR UPDATE。
- **樂觀鎖 (Optimistic Lock)** — 先假設不會撞,事後用版本欄位檢查有沒有人搶先改過;沒撞就過,撞了就重試。適合衝突很少見的場景,不必事先付鎖的代價。
  - 想更深可以想想:關鍵字:optimistic concurrency control、compare-and-swap。
