## 為什麼要先自己搞出一次 bug

L0 說了超賣是「同時」惹的禍,但光用聽的很難有感覺。這課讓你在自己的電腦上,
親手重現阿哲那晚發生的事——**一個簡化到極致的模型**:先不碰真正的資料庫,
用一個 Python 變數假裝「資料庫裡的庫存欄位」,50 個「顧客」同時搶購。

## 動手:寫一支會超賣的搶購腳本

貼進一個 `.py` 檔案跑跑看:

```python
import threading
import time

stock = 10   # 假裝這是資料庫裡「手作包」這一列的庫存欄位
sold = 0

def buy(customer_id):
    global stock, sold
    current = stock              # ① 讀庫存
    time.sleep(0.01)             # 模擬「判斷、呼叫金流」耗掉的一點點時間
    if current > 0:              # ② 判斷還有沒有貨
        stock = current - 1      # ③ 扣庫存
        sold += 1

customers = [threading.Thread(target=buy, args=(i,)) for i in range(50)]
for c in customers:
    c.start()
for c in customers:
    c.join()

print(f"庫存只有 10 件,卻成交了 {sold} 筆")
print(f"最終庫存變數:{stock}(應該停在 0,不該是負的)")
```

跑起來,你會看到類似這樣的結果:「庫存只有 10 件,卻成交了 47 筆」,
最終庫存變成負數。50 個顧客,10 件貨,超賣一大半。

## 剛剛發生了什麼

看 `buy()` 函式的三步:**讀 → 判斷 → 寫**。中間那行 `time.sleep(0.01)`
不是搗亂用的道具——它模擬的是真實世界一定會有的耗時:判斷邏輯、呼叫金流、
組裝回應…現實中的「讀庫存」和「扣庫存」之間,永遠隔著一段不是零的時間。

問題就出在這段空隙:50 個顧客幾乎同時執行到「①讀庫存」,那一瞬間他們讀到的
`current` 全部都是 10(因為誰都還沒來得及扣)。空隙一過,50 個人全部
以為「還有貨」,於是全部把庫存減 1。這種**先檢查、再動作**的寫法叫
**Check-Then-Act**——只要檢查和動作之間有任何空隙,空隙裡發生的事就可能
讓檢查結果過期失效。

這正是 **race condition** 的真面目:程式邏輯完全正確
(讀、判斷、扣,每一步都沒寫錯),錯的是**沒人保證這三步是一整塊、不會被打斷**。

## 常見坑

| 症狀 | 原因 |
|---|---|
| 自己單獨跑一次,完全正常 | 沒有其他人「撞」進同一個空隙;正式環境人一多,空隙才會被撞到 |
| 加一行 print 進去觀察,bug 就消失了 | print 本身耗時,意外改變了時間差,讓空隙變窄甚至消失——這是除錯這類 bug 最容易踩的陷阱 |
| 把 sleep 拿掉,問題好像不見了 | 空隙變超短,機率降低不代表沒有;正式環境的耗時(呼叫金流、序列化)一樣存在,只是你沒故意放大它 |
| 覺得「機率很低就算了」 | 秒殺當晚正是「一堆人同時撞進同一個空隙」機率最高的時刻,低機率事件會被流量放大成常態 |

## 📇 名詞卡

- **Check-Then-Act** — 先檢查某個條件(還有庫存嗎?),再根據檢查結果採取動作(扣庫存)。只要檢查和動作之間存在任何時間差,這個時間差裡發生的事就可能讓檢查結果過期——這是幾乎所有併發 bug 的共同根源。
  - 想更深可以想想:下一課的三種修法,本質上都是在消除這段時間差。
- **Race Condition(競態條件)** — 程式的每一步邏輯都沒寫錯,但結果卻取決於「誰先誰後」這種時間上的巧合。超賣、重複扣款、髒讀都是它的常見面孔。
  - 想更深可以想想:維基百科 Race condition 條目有更完整的分類(資料庫、作業系統、硬體層級各有例子)。
