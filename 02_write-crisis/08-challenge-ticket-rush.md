# L8|放手課:設計一場百萬人搶票 🎓

🎯 這課結束時:你會設計出一套「演唱會 5000 張票、上百萬人同時搶」的下單架構,並能為每個元件說出它解決了哪一課教的哪個具體問題。
🧩 需要先會:本模組全部(L1–L7)——這課不教新東西,只給你機會把學過的每一把武器組裝起來。
📚 想深挖:無新素材。回頭挑一課覺得還不夠熟的,重讀一次再回來動手。

## 情境

某天阿哲接了一個新案子:一場演唱會,只有 5000 張票,預估開賣瞬間會湧入
上百萬人搶購。他丟給你一句話:「跟秒殺很像對吧?你來設計。」

## 任務

設計一套下單架構,處理「開賣瞬間上百萬人同時搶 5000 張票」這件事。
這課不給你一步一步的做法——你自己決定要用本模組哪些武器、怎麼組裝。

## 驗收條件

不管你的架構長什麼樣子,它必須滿足:

1. **絕對不超賣**:5000 張票,賣出的數量精準等於 5000,一張不多。
2. **每個人都會知道結果**:搶到或沒搶到,就算不是按下去那一秒立刻知道,
   最終也要有明確的方式讓使用者查到自己的結果。
3. **尖峰有被削平**:開賣瞬間的請求不會直接、全部衝進資料庫。
4. **你能為每一塊說出「為什麼」**:架構裡的每個元件,你都答得出來
   ——它是為了解決前面哪一課提到的哪個具體問題,而不是「大家都這樣做」。

## 預設路徑(卡住的話用這個起頭)

一份寫了一半的架構描述,你來把空格填完:

```
使用者搶票請求
  → CDN / 負載平衡(擋掉靜態資源,只讓真正的搶票請求進來)
  → 排隊頁面 / ______________(攔住開賣瞬間的尖峰,不讓它直衝資料庫)
  → 消費者依序處理每一個排隊中的請求:
       ① 確認這位使用者還沒搶過票(防止重複下單)
       ② ______________
          (這裡要填:怎麼保證同一張票不會被兩個人同時搶走?
          從 L2 的三種修法裡選一種,說出你選它的理由)
       ③ 寫入訂單,標記這個人「搶到」或「沒搶到」
  → 通知使用者結果:______________
     (立即輪詢查詢結果?還是模組三會教的即時推播?先寫下你的選擇就好)

資料庫設計:
  票只有 5000 張,資料量非常小 —— 這裡 ______________
  (要不要分片?為什麼?提示:分片解決的是「資料量」還是「同時搶的人數」
  這兩個問題,先想清楚兩者是不是同一件事)
```

**允許 **降級交付****:如果一次做不出完整版,先做出
「單機、絕不超賣」的最小版本(只用 L2 的原子 UPDATE 擋住超賣,不管佇列與
擴充),確定這一層穩了,再逐步疊加削峰與其他元件——這樣分階段完成,
一樣算是完成了這次挑戰。

## 自我驗收

設計完之後,自己對照這張表檢查一遍(不必交給任何人,自己說得出來就算過):

- [ ] 同一張票不可能同時賣給兩個人——我能具體指出是哪一種鎖或原子操作擋住的
- [ ] 開賣瞬間上百萬個請求同時湧入,資料庫不會被直接打爆——我能指出削峰的機制在哪裡
- [ ] 每個使用者最終都會知道自己搶到了沒有,就算不是當下立刻知道
- [ ] 架構裡任何一塊拿掉,我都能說出會出什麼問題(如果有一塊拿掉也沒差,那它可能根本不需要)
- [ ] 我想清楚了「這次到底要不要分片」,而不是看到「大流量」就反射性搬出所有武器

→ 下一課:把這趟從超賣到搶票的旅程,自己講一遍,再考自己幾題。

## 📇 名詞卡

- **降級交付 (Degraded Delivery)** — 先做出功能較少但核心正確的最小版本(例如只保證不超賣),確定穩了之後再逐步疊加其他能力,而不是一次要求做出完整版才算數。
  - 想更深可以想想:系統設計練習與真實專案都常用這個策略,避免因為求全而遲遲交不出東西。
