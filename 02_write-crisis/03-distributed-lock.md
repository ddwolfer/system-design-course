# L3|分散式鎖:鎖要鎖在誰都看得到的地方 📖

🎯 這課結束時:你知道「程式內的鎖」為什麼在多台機器的世界裡失靈,以及 Redis 分散式鎖的原理、代價與該不該用它。
🧩 需要先會:L2 的三種修法(尤其是悲觀鎖的概念)。
📚 想深挖:Redis 官方文件「SET」指令頁(NX、EX 選項);關鍵字:distributed lock、Redlock、lease。

## 訂單變多之後,新的協調問題出現了

阿哲的網站扛過了超賣危機,訂單量也跟著漲了。出貨流程原本是一台背景程式
慢慢處理排隊的訂單、呼叫物流公司 API 產生運單。訂單一多,他把這支背景程式
複製成三份同時跑(三台 worker 分攤工作)。

新問題來了:如果 worker A 和 worker B 同時撈到「同一張訂單」去處理,
兩邊都會呼叫一次物流 API——結果是同一張訂單生出兩張運單,多付一次運費,
倉庫也搞不清楚該包哪一箱。

## 為什麼不能只在程式裡上個鎖就好

如果 worker 只有一台,在程式裡用一般程式語言內建的鎖(例如 Python 的
`threading.Lock()`)確實能擋住「同一個程式裡的兩段程式碼同時處理同一張訂單」。
但現在是**三台各自獨立的機器**,各自的記憶體互不相通——A 機器上的鎖,
B 機器完全看不到,鎖了跟沒鎖一樣。

L2 的資料庫鎖(`SELECT ... FOR UPDATE`)之所以在多台 app server 之間依然有效,
是因為鎖真正的位置在**資料庫**——所有機器共用同一顆資料庫,鎖自然看得到彼此。
但「呼叫物流 API 只呼叫一次」這件事本身不是一行資料庫欄位,沒辦法靠一句
`UPDATE ... WHERE` 解決。這時候才輪到**分散式鎖**出場:把「誰現在有權處理
這張訂單」這件事,交給一個**所有機器都連得到的共用儲存**來仲裁。

## Redis 分散式鎖的白話原理

最常見的做法是拿 Redis 的一句指令當鎖:

```
SET order-lock:1024 worker-B NX EX 30
```

拆開看:

- `SET order-lock:1024 worker-B`:想在 Redis 裡寫一個 key 叫
  `order-lock:1024`,值是「worker-B」(記錄是誰拿到鎖)。
- `NX`(**Not eXists**):**只有這個 key 目前不存在時才會寫入成功**。
  如果 worker-A 已經搶先設定過,worker-B 這句會直接失敗——這就是「搶鎖」的機制:
  誰先讓 Redis 記下自己的名字,誰就拿到鎖,其他人自動落空。
- `EX 30`:這個 key **30 秒後自動消失**(**TTL,存活時間**)。

## 為什麼一定要有 TTL

假設沒有 `EX 30`:worker-B 拿到鎖之後,如果那台機器當機、網路斷線、
或程式直接掛掉——它再也沒機會去刪掉那把鎖。這張訂單就永遠卡在
「有人在處理」的狀態,沒有人敢碰,永遠出不了貨。

TTL 就是保險絲:就算持鎖者真的死掉,30 秒後 Redis 會自動把鎖收回,
讓別的 worker 能重新接手。代價是:如果處理這張訂單真的需要超過 30 秒,
鎖會在工作做完前就自動消失,得靠「續租(定期延長 TTL)」來處理——
這也是分散式鎖真正麻煩的地方,見下面的但書。

## 釋放鎖要先驗身分

worker 處理完訂單,理當把鎖刪掉讓別人可以用。但**不能直接無腦刪**:

想像 worker-B 拿到鎖後,處理得比較慢,30 秒 TTL 到了,Redis 自動把鎖收回;
這時 worker-C 搶到了同一把鎖,開始處理同一張訂單。就在這時候,
worker-B **終於**處理完,執行「刪除鎖」——如果它不檢查鎖現在是誰的,
就會把 worker-C 剛拿到的鎖誤刪,導致又有第三個 worker 也搶得進來,
一張訂單被多個 worker 同時處理,鎖形同虛設。

正確做法是刪除前先確認:「Redis 裡這把鎖的值,還是不是我當初寫的
`worker-B`?」如果不是,代表自己的鎖早就過期被別人拿走了,不該刪。
(這一步本身也要做成一個不可分割的動作,否則又會撞回 L1 那個
check-then-act 的老問題——正式做法通常用一小段 Lua 腳本讓
「檢查值、刪除」在 Redis 裡一次做完,不留空隙。)

## 誠實的但書:自製分散式鎖,坑比想像中多

分散式鎖聽起來簡單(`SET ... NX EX` 一行搞定),實際上暗礁不少:

- **時鐘問題**:TTL 靠的是時間,如果不同機器的系統時間有落差,
  「30 秒」在每台機器上算出來可能不一樣。
- **續租的複雜度**:長任務要在鎖過期前主動延長 TTL,延長本身又是一次
  「檢查是不是還是我的鎖」的操作,又要處理各種邊界情況。
- **網路分區**:持鎖者其實還活著、只是暫時斷線,Redis 誤判它死掉把鎖收回,
  結果變成兩個人同時以為自己是唯一持鎖者(這正是模組二最後一課 CAP
  要講的物理限制)。

**因此:分散式鎖是最後手段,不是第一選擇。** 能用 L2 那種資料庫原子操作解決的,
優先用資料庫;真的需要跨機器協調、又繞不開的場景,才上分散式鎖,而且
盡量用像 Redlock 這類經過驗證的現成演算法或函式庫,不要自己從零手刻。

## 📇 名詞卡

- **TTL (Time To Live)** — 一份資料在儲存系統裡的存活時間,時間一到就自動消失。分散式鎖靠它當保險絲——就算持鎖者當機忘記解鎖,鎖也會自動釋放,不會卡死系統。
  - 想更深可以想想:Redis 文件:EXPIRE / SET 指令的 EX 選項。
- **NX (Not eXists)** — Redis SET 指令的選項:只有這個 key 目前不存在時,這次寫入才會成功。多個 worker 同時搶著 SET 同一個 key,只有一個人會成功——這正是分散式鎖「搶鎖」的機制。
  - 想更深可以想想:Redis 官方文件 SET 指令頁,NX / XX 選項說明。
