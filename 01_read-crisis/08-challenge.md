# L8|放手課:三個病人,你來開藥方 🎓

🎯 這課結束時:你會拿到三個「網站慢」的真實情境,只給症狀和驗收條件——由你自己判斷該用哪把刀(或哪幾把刀組合),並說得出取捨代價。
🧩 需要先會:L3–L7 全部(索引、快取、讀取副本、CDN/負載平衡、量級直覺與決策樹)。
📚 想深挖:沒有標準答案讀物——這課是拿前面學的東西練手,深挖回頭看 L3–L7 各自的📚。

## 規則

三個情境,各自只給「症狀」和「驗收條件」,不給答案。你可以只做其中一個,
也可以三個都挑戰——**只完成情境 a 也算過關**,重點是診斷的過程站不站得住腳,
不是做了幾題。每題自我驗收時,問自己三件事:

1. 你的診斷依據是什麼(你會怎麼證實它,不是憑感覺)?
2. 你開出的藥方是什麼(可以不只一把刀)?
3. 這帖藥方的代價是什麼(沒有免費的午餐)?

## 情境 a:部落格文章頁很慢

「好物市集」加了一個部落格(分享皮革保養知識),流量不大,但**每個人看到的
內容都一樣**——同一篇文章,一千個人點閱,每次都重新跑一次資料庫查詢
組出同一頁 HTML。你查了一下這句查詢的 **EXPLAIN 查詢計畫**,
發現它在用 Seq Scan 找文章。

**驗收條件**:說出至少一個診斷依據、開出的藥方、以及這帖藥方的代價。

## 情境 b:商品搜尋結果頁,尖峰時很慢

尖峰時段(晚上八點到十點),「輸入關鍵字查商品」這個功能回應變得很慢。
你發現同一個熱門關鍵字(像「皮夾」)在同一分鐘內被搜尋了幾千次,
每次都是一句完整的資料庫查詢。

**驗收條件**:說出至少一個診斷依據、開出的藥方(留意:這題的藥方**不只
一種刀能解**,想想不同藥方各自的取捨)、以及代價。

## 情境 c:全球用戶圖片載入很慢

「好物市集」商品圖片放在阿哲那台位於台灣的伺服器上。歐洲和美洲的使用者
反映商品圖片要等好幾秒才跑出來,但台灣本地使用者完全沒感覺。

**驗收條件**:說出至少一個診斷依據、開出的藥方、以及代價(這帖藥方是不是
對所有類型的內容都適用?)。

## 預設路徑:診斷表格範本

沒把握從空白開始的話,照這張表填(已經幫你填了一部分當範例):

| 情境 | 你觀察到的症狀 | 懷疑的原因 | 怎麼驗證 | 對應的刀 |
|---|---|---|---|---|
| a | 同一篇文章,每次都重新查、重新組頁 | 內容重複度高,查詢本身也可能沒優化 | 先 EXPLAIN 看有沒有 Seq Scan | 待你填:索引?快取?還是兩者都要? |
| b |  |  |  |  |
| c | 只有遠端使用者慢,本地使用者正常 | 距離造成的網路延遲,不是伺服器算力問題 |  |  |

情境 b 整列留白,自己觀察、自己填。

## 常見坑

| 症狀 | 原因 |
|---|---|
| 一律開「加索引」當萬靈藥 | 沒先用 EXPLAIN 確認問題真的出在查詢方式,索引解決不了「重複問同一件事」或「距離太遠」的問題 |
| 把快取當成資料庫的替代品 | 忘了快取只是「暫存」,原始資料還是得有地方存、有辦法查到最新版 |
| 只講出一把刀,沒講代價 | 每帖藥方都有取捨(錢、複雜度、資料新鮮度),沒講代價等於答案只講了一半 |

## 📇 名詞卡

- **查詢計畫(Query Plan)** — 資料庫執行一句查詢前,會先決定「怎麼找資料」的步驟——用 EXPLAIN 這類指令能把這個計畫印出來看,判斷它是不是在用 Seq Scan(整表翻)這種昂貴方式。
