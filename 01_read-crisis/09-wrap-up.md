## **回講**:自己把故事講一遍

找一張紙或開一個新的空白畫面,不看任何一課筆記,試著把整個模組講給
「不懂技術的朋友」聽——就當作阿哲又打來問了。這種不看筆記、講給別人聽的
練習就是 **回講法**,建議照這個順序:

1. 那晚到底發生了什麼、為什麼「程式碼沒動網站卻掛了」。
2. 一個請求原本要跑哪些站(client、DNS、server、資料庫、回程)。
3. 四把刀分別解決了什麼問題、砍下去的順序為什麼是這樣。
4. 如果流量再翻十倍,你會先檢查哪裡。

講得卡住的地方,就是你這個模組還沒真正吃透的地方——回頭翻那一課,
而不是硬背過去。

## 隨堂考(先想,再看答案)

讀完題目**先暫停,自己想過一輪**,想不出來也沒關係,想完再往下看答案。

**Q1.** 資料庫查詢加了索引之後還是全表掃描(Seq Scan),最可能的原因是什麼?

答案:查詢條件對欄位做了運算(例如 `WHERE lower(name) = ...`),而索引是建在
「原始欄位」上,不是「運算後的結果」上——資料庫用不上那個索引。

**Q2.** 快取的 TTL 到期瞬間,大量請求同時湧入資料庫,這個現象叫什麼?
可能的應對方向是什麼?

答案:cache stampede(快取雪崩/驚群)。方向包括:提前續期(還沒到期就先
背景更新)、只讓一個請求去補資料其他人等待、或錯開不同 key 的到期時間。

**Q3.** 讀寫分離架構下,使用者下單後馬上看不到自己的訂單,這個現象的名稱
和至少一個處理方向是什麼?

答案:read-your-own-writes 問題。處理方向包括剛寫入後短時間內相關讀取
強制走 primary、或副本自己追蹤複寫進度必要時延後讀取。

## 深讀地圖

這模組每一課都刻意留了「量級」而不是精確答案,想補完整,去讀官方文件:

- **索引**:PostgreSQL 官方文件 Chapter 11「Indexes」——尤其是 Index Types
  和 Using EXPLAIN 兩節。
- **快取**:Redis 官方文件的 Caching 專章——cache-aside、TTL 策略、
  stampede 的官方建議解法。
- **CDN 與負載平衡**:任一雲廠商的 CDN 與 Load Balancer 架構總覽文件——
  重點看「哪些內容適合放邊緣節點」和「健康檢查怎麼設計」。
